# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

WindowGingerPaw is the Windows port of **GingerPaw** (macOS codename FlowOSS), an on-device push-to-talk dictation app. The original macOS app lives in a sibling repo at `..\gingerpaw` — that repo is the source of truth for behavior, state machine, and prompt text being ported here. This repo shares no code or build tooling with it; it's a from-scratch .NET implementation of the same product.

Core loop: hold a hotkey → record mic audio → on release, transcribe locally with Whisper → optionally clean up the transcript with a local Qwen2.5-0.5B-Instruct LLM → paste into the focused app via clipboard + synthetic Ctrl+V.

No model training happens anywhere in this project. Both Whisper and Qwen are pretrained, quantized, downloaded open-weight models — the only ML work here is loading and prompting existing weights.

See `plan.md` in this same directory for the full phased build plan (setup, model acquisition, phase-by-phase build order, testing strategy, ranked risks).

## Architecture mapping (Mac → Windows)

| Mac (FlowKit) module | Windows project | Key OS primitive |
|---|---|---|
| `Settings` | `GingerPaw.Settings` | JSON file at `%AppData%\GingerPaw\settings.json` |
| `Permissions` | `GingerPaw.Permissions` | Mic privacy only — no Input Monitoring/Accessibility equivalent exists on Windows |
| `Audio` | `GingerPaw.Audio` | NAudio/WASAPI capture, 16kHz mono PCM WAV |
| `Hotkeys` | `GingerPaw.Hotkeys` | `WH_KEYBOARD_LL` via `SetWindowsHookEx` — default key **Right Ctrl** (Fn is unreliable on Windows; usually consumed by keyboard firmware) |
| `Transcription` | `GingerPaw.Transcription` | `Whisper.net` (whisper.cpp binding) over a GGUF Whisper model |
| `TextProcessing` | `GingerPaw.TextProcessing` | `LLamaSharp` (llama.cpp binding) over a GGUF Qwen2.5-0.5B-Instruct model |
| `TextInsertion` | `GingerPaw.TextInsertion` | `SendInput` (Ctrl+V) + `Clipboard` |
| `Dictation` | `GingerPaw.Dictation` | Ported near-verbatim — same state machine, same method names |
| `Overlay` | `GingerPaw.Overlay` | Borderless `Topmost` WPF window |
| `AppCore` | `GingerPaw.App` | Composition root + tray icon (`H.NotifyIcon.Wpf`) + settings window |

Every module wraps one OS-specific primitive behind an interface (`IAudioRecording`, `ISpeechTranscriber`, `ITextInserter`, `ITextProcessor`), mirroring the protocol-seam pattern the Mac app uses for testability — `GingerPaw.Dictation` should be unit-testable with hand-written stubs, exactly like `DictationCoordinatorTests.swift` on the Mac side.

## Status

**Phase A (done):** `GingerPaw.Hotkeys` (Right Ctrl low-level keyboard hook), `GingerPaw.Audio` (NAudio `WaveInEvent`, direct 16kHz mono capture — no manual resampler), `GingerPaw.Transcription` (Whisper.net + `WhisperModelProvisioner` auto-downloads `ggml-base.bin` into `models/whisper/`), `GingerPaw.TextInsertion` (Clipboard + 4-event `SendInput` Ctrl+V — mind the `INPUT` struct's union must include `MOUSEINPUT`/`HARDWAREINPUT`, not just `KEYBDINPUT`, or `SendInput` silently fails to match the OS ABI size). The original throwaway console `Program.cs` has been deleted — `GingerPaw.App` is now the real Phase B app.

**Phase B (done):** `GingerPaw.Settings` (JSON store at `%AppData%\GingerPaw\settings.json`), `GingerPaw.Permissions` (mic-only check via the Windows privacy registry key), `GingerPaw.Dictation` (`DictationCoordinator` ported as a C# record hierarchy for `DictationState`; 4 ported unit tests in `tests/GingerPaw.Dictation.Tests` all pass). `GingerPaw.App` is now a windowless WPF tray app (`OutputType=WinExe`, `UseWPF=true`, `ShutdownMode=OnExplicitShutdown`) — `AppComposition`/`AppRuntime`/`TrayController` mirror the Mac's composition root, using `H.NotifyIcon.Wpf` **pinned to 2.3.2** (2.4.1+ dropped .NET 8 support, only targets net10/.NET Framework). Hotkey callbacks are marshaled onto the WPF `Dispatcher` via `BeginInvoke` so `Clipboard` calls land on the STA UI thread correctly.

**Phase C (done):** `GingerPaw.Overlay` — borderless/topmost/transparent WPF pill window (`PillOverlayWindow`) mirroring the Mac's `DictationOverlayController.swift`: paw glyph + purr pulse while Recording, 3-bar pulse while Processing, download glyph while Inserting, checkmark + bounce on Copied. **Intentional divergence from Mac:** the pill also shows on `Failed` (red, warning glyph, bounce) instead of staying hidden like the Mac source does — a visible failure cue was judged more useful while this port is still being hardened. Icons are Segoe MDL2 Assets codepoints (`U+E720` mic, `U+E896` download, `U+E8FB` checkmark, `U+E783` warning) — the Windows analog of the Mac's SF Symbols. Note: these codepoints render as blank/invisible in plain-text tooling (grep, file read, terminal echo) since they're Private-Use-Area characters with no glyph in most fonts — always write them via `\uXXXX` C# escapes, never as literal pasted characters, or they silently get mangled. `OverlayPositioning.ComputeTopLeft` is a pure, unit-tested function (`tests/GingerPaw.Overlay.Tests`, 3 tests) mirroring the Mac's bottom-center placement math. Wired into `AppRuntime`'s `OnStateChange` alongside the tray update; gated on the existing `GingerPawSettings.ShowPill` (already defaulted `true`, no settings UI yet to toggle it — edit `%AppData%\GingerPaw\settings.json` directly if needed). Manually verified working end-to-end, including the Failed-state red flash.

**Phase D (done):** `GingerPaw.TextProcessing` — `LLamaSharpTextProcessor` (LLamaSharp 0.27.0 + `LLamaSharp.Backend.Cpu`, `StatelessExecutor` over a GGUF Qwen2.5-0.5B-Instruct model auto-downloaded by `QwenModelProvisioner` into `models/qwen/` — `Qwen2.5-0.5B-Instruct-Q4_K_M.gguf` from `bartowski/Qwen2.5-0.5B-Instruct-GGUF`, deferred until the first `FormatAsync` call rather than at app startup, since `FormatEnabled` defaults `false`). `PromptBuilder`/`OutputCleaning`/`OutputValidation` are pure, unit-tested functions (`tests/GingerPaw.TextProcessing.Tests`, 17 tests). Three real bugs found and fixed during manual testing, all worth knowing about before touching this code again:
  1. **Runaway repetition** — the model would loop the same answer 15+ times until hitting the token cap. Fixed with `RepeatPenalty`/`PenaltyCount` on `DefaultSamplingPipeline` plus an explicit stop-marker truncation as a belt-and-suspenders on top of `AntiPrompts` (streaming antiprompt detection in LLamaSharp can still leak partial matches through).
  2. **Wrong prompt format (the real root cause behind the worst failures)** — Qwen2.5-Instruct is trained on the ChatML turn format (`<|im_start|>`/`<|im_end|>`), and `PromptBuilder` originally built a plain-text completion prompt instead. A small instruct model fed the wrong format doesn't reliably know it's mid-task at all — symptoms ranged from echoing its own rule text back as content to falling back to generic "I am an AI assistant..." boilerplate. Fixed by rebuilding the prompt as real user/assistant ChatML turns.
  3. **Mac's original 6-example/7-rule prompt was too long for a 0.5B model** — cut down and rebalanced to 5 short examples covering numbered-list/bulleted-list/passthrough/question patterns (shorter than Mac's, but more pattern coverage than the first 2-example cut tried — each example just stays short). `OutputValidation.LooksValid` is a safety-net fallback: if formatted output still looks broken (implausibly longer than the input, or contains phrases suggesting prompt-leakage), the original raw transcript is pasted instead — pasting exactly what was said matters more than getting restructuring right every time the model is this small.

  Remaining known quality gap: the model doesn't always catch every ordered-list phrasing (e.g. caught "first... then... finally" but initially missed "the first thing is X, second is Y..." until a closer-matching example was added) — bulleted vs. numbered list choice for ordered sequences is also inconsistent. Good enough to ship as "sometimes restructures, never corrupts," not perfect.

**Settings window, permission flow, and autostart (done):** `GingerPaw.App/SettingsWindow.xaml(.cs)` — a real settings UI at last (previously only reachable by hand-editing `settings.json`), opened via a new "Settings…" tray menu item, with a Settings tab (auto-paste/restore-clipboard/AI-formatting/show-pill/launch-at-startup checkboxes, each writing straight through to `GingerPawSettings` on toggle) and a Permissions tab (mic status + a button calling `PermissionCenter.OpenMicrophoneSettings()`, which existed but was never wired to anything before this). `GingerPaw.Settings/AutostartManager.cs` is new — registry-based (`HKCU\...\Run`), the registry itself is the single source of truth (no duplicate settings.json field). `AppRuntime.Start()` now also checks mic permission at startup and shows a tray balloon (`TaskbarIcon.ShowNotification`, needs the `H.NotifyIcon.Core` namespace for `NotificationIcon` — not auto-imported alongside `H.NotifyIcon`) if access is `Denied`; `Unknown` is left alone since that's just "Windows hasn't asked yet," not actionable. This was pulled forward from Phase E (`plan.md`'s "Add a LaunchAtStartup setting" item) since it's tightly coupled to the same settings window. Manually verified: window opens from tray, all toggles persist across reopen, mic status displays correctly.

**Known-good, manually verified:** tray icon registers (`TaskbarIcon.IsCreated == true`) but appears in Windows' hidden-icons overflow by default, not the visible taskbar — check the `^` chevron near the clock. The full hold-Right-Ctrl → record → transcribe → format → paste loop works end-to-end inside the WPF tray app, confirmed after fixing a real bug in `NAudioRecorder.StopCapture()` — it disposed the WAV writer inside a `RecordingStopped` event handler banking on subscription-order timing to beat a separate wait signal; that race lost every time, leaving the file locked and every transcription failing with "file in use." Fixed by waiting for `RecordingStopped` then disposing the writer explicitly itself, with no order dependency. A temporary `App.Log(...)` helper (mirrors every line to both console and `%TEMP%\gingerpaw-startup.log`) was added to `App.xaml.cs`/`AppRuntime.cs` for this debugging, plus a temporary `DictationCoordinator.LastRawTranscript` property exposing the pre-format transcript for comparison against the formatted one — **delete both logging mechanisms once Phase D's output quality is settled**, per the `TEMPORARY DIAGNOSTIC` markers left in those files.

**UI polish pass (mostly done):** Driven by the user noticing the app "doesn't look like the one built for Mac" — paused Phase E packaging to fix this first. Covered three areas:
  1. **App icon (done)** — `src/GingerPaw.App/Assets/AppIcon.ico` built from the Mac app's `AppIcon.appiconset` PNGs (multi-res, PNG-compressed ICO entries at 16/32/64/128/256px). Wired via `<ApplicationIcon>` in `GingerPaw.App.csproj` (exe icon), `TrayController.cs`'s `TaskbarIcon.IconSource` (a `BitmapImage` from the `pack://application:,,,/Assets/AppIcon.ico` URI — **use `IconSource`, not the legacy `Icon` property**: GDI+'s classic icon loader can't reliably decode this file's PNG-compressed frames, WIC can), and `SettingsWindow.xaml`'s `Window.Icon`. All three manually verified showing the correct icon.
  2. **SettingsWindow restyle (done)** — rebuilt from a plain `TabControl` into a dark-themed, card-based layout closely mirroring the Mac's `SettingsView.swift`/`UIComponents.swift`: a left sidebar with icon+label nav items (Settings/Permissions, replacing the Mac's `FlowSection` sidebar list) instead of native tabs, card sections per settings group (OUTPUT/AI FORMATTING/APPEARANCE/STARTUP) with 12px rounded corners + border, a custom-templated checkbox and button using the Mac's dark-mode violet accent (`#A37CEF`, from `Brand.accent` in `UIComponents.swift`), and a colored capsule for microphone permission status mirroring the Mac's `StatusPill`. Window is now resizable (`ResizeMode="CanResize"`, 600×620 default, 560×480 minimum) with a working `ScrollViewer` (`VerticalScrollBarVisibility="Auto"` — it was previously `Disabled`, which silently clipped the Startup card off the bottom of the window with no way to reach it) and a custom thin dark scrollbar style (the default WPF one is white-on-white against a dark background). One real bug hit and fixed along the way: the sidebar `RadioButton`s have `IsChecked="True"` set in XAML for the default-selected item, which fires the `Checked` event *during* `InitializeComponent`'s BAML parse, before later-declared named fields (`SettingsPage`, `HeaderTitle`, etc.) are assigned — threw a `NullReferenceException` on every Settings-window open. Fixed with a null-guard at the top of both nav `Checked` handlers in `SettingsWindow.xaml.cs`.
  3. **Tray menu / app chrome consistency (not started)** — wording/structure review of `TrayController.cs`'s context menu still pending.

**Not yet verified:** settings-persistence-across-restart (the *file* persists fine — confirmed via direct JSON edits all session — but toggling via the new Settings window and then fully restarting the app hasn't been explicitly re-checked). Multi-monitor/mixed-DPI overlay placement (`OverlayController` only targets the primary screen's `SystemParameters.WorkArea` for now).

## Conventions to carry over from the Mac app

- Keep method/type names close to the Swift originals where sensible (`StartRecording`/`StopRecordingAndProcess`/`CancelRecording`, `DictationState`, `InsertionOutcome`) — makes porting and side-by-side review easier.
- Anything behind an interface seam gets an xUnit test with a stub. Raw OS-boundary code (the keyboard hook, WASAPI, `SendInput`, tray icon) is manual-only by nature — don't invent fake unit coverage for it.
- Settings default to the same values as the Mac app where applicable (e.g. `FormatEnabled` defaults to `false`).
